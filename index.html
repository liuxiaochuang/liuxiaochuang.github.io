OSI参考模型

  • 物理层：传输比特流，有多种接口标准，网线接口就是其中之一
  • 传输层：传输帧，有成帧、差错控制、流量控制、传输管理功能。
  • 网络层：传输数据报，关键功能是路由选择，并实现拥塞控制、流量控制、差错控制、网际互连。该层协议有IP、IPX、ICMP、ARP、OSPF等
  • 传输层：传输报文段（TCP）或者数据报（UDP)，进程通信，提供流量控制、差错控制、服务质量等功能。
  • 会话层：session，建立不同主机各进程上的会话，实现有序数据传输
  • 表示层：信息交换方式，数据压缩，加密，等
  • 应用层：

操作系统

Veröffentlicht am 2018-03-17 | in 技术

用户态、内核态和系统调用

  • 内核态：就是操作系统内核完成的工作，具体上是指执行的指令，包括最底层的时钟管理、中断处理、设备驱动和使用频率较高的程序如进程管理、存储管理（IO）、设备管理等。内核态的CPU指令优先级更高。
  • 用户态：非内核态的指令。
  • 系统调用：就是用户态的代码要去执行内核态的指令，通过系统调用执行。linux下的公用库函数则是将多个系统调用封装起来了。
  • 用户态切换为内核态的三种情况：系统调用（也是中断，软中断，程序主动要求的），异常（缺页），中断

进程

  • PCB，进程控制块、进程的唯一标志
  • 进程的五种状态和转换（创建、就绪、执行、阻塞、结束），操作系统要调度这些进程。
  • 进程性质：动态性（状态改变），并发性（多个进程并发执行），独立性（进程之间互不干扰）

线程

  • 多进程的目的是为了提高资源利用率，但是进程太多会使系统将大量时间用在进程切换上。所以有了线程，线程的切换消耗的资源相对较少。
  • 线程拥有的资源:线程id、程序计数器、寄存器和堆栈。

所以多线程要比多进程要有更高的并发性。

  • 线程模型：n个用户线程对应m个内核线程，n>=m；

进程调度

  • 概念：在进程的就绪队列中，按照一种公平的算法，将cpu分配给它执行，以实现进程的并发执行。
  • 类型：先来先服务、短作业优先、优先级调度法高相应比优先调度（（wait时间+cpu时间）/cpu时间）、时间片轮转算法。

进程同步

  • 临界资源：许多硬件资源如打印机都是临界资源。
  • 临界区：访问临界资源的代码是临界区。进程对于临界区的访问要互斥。
  • 实现临界区互斥方法：软件方法、硬件方法、信号量方法（PV操作，本科经常考的那种）
  • 原语：完成某个功能不被分割不被中断的序列。
  • 经典问题：读写问题、哲学家就餐问题。
  • 死锁：多进程因竞争资源造成的相互等待状态。
  • 死锁必要条件：互斥访问、保持并占有、不可剥夺、循环等待。不是充分条件。
  • 死锁的预防、避免（银行家算法）、检测、解除。

内存管理

  • 概念：操作系统对内存的划分和动态分配
  • 内存管理内容：内存空间的分配和回收、地址转换、内存扩充（虚拟存储技术、自动覆盖技术）、存储保护（防止进程越界）
  • 运行程序：创建进程首先要将程序和数据装入内存，将用户的源代码变为内存中的程序需要：编译->链接->装入内存
    • 链接方式：静态链接（目标文件装入内存前链接）、装入使动态链接（边装入边链接）、动态链接（运行时链接，so文件，便于共享）
    • 装入方式：绝对装入（逻辑地址和物理地址相同）、可重定位装入\静态重定位（一次性全装入内存）、动态重定位（一开始装入一部分，后来再动态申请内存装入）
  • 扩充内存：覆盖内存、交换（把等待状态的内存移到辅存）
  • 内存的连续分配管理：单一连续分配（单进程）、固定分区分配（给进程固定物理地址）、动态分区分配（进程装入内存时，根据大小动态建立分区，算法有首次适应、最佳适应、最坏适应、临近适应）
  • 内存的非连续分配管理：
    • 分页管理：页表、逻辑空间、物理空间、块表、二级页表、页高速缓存（cache）
    • 分段管理：段号、段内偏移、二维地址
    • 段页式管理：段+页
  • 虚拟存储管理：
    • 局部性原理：
      • 时间局部性
      • 空间局部性
      • 相关技术：块表、页高速缓存、虚拟存储技术。
    • 虚拟存储技术：
      • 程序先将一部分装入内存，然后运过程中将程序另一部分装入，这样好像内存变大了
      • 操作系统将暂时不用的内存存入外存（外存、页表机制、中断机制、地址映射）。
        • 页面置换算法（减少缺页率）：最佳置换算法（理想的算法，无法实现）、FIFO、LRU算法、时钟置换算法
        • 页面分配策略：
          • 固定分配局部置换
          • 可变分配全局置换
          • 可变分配局部置换
        • 页面调入时机：
          • 预调页策略
          • 请求调页策略
        • 抖动：调入又调出页
        • 工作集：某段时间间隔内，进程要访问的页面集合。
        • 地址翻译：外<-->内页转换

文件管理

  • 用户的I\O就是以文件为基本单位的，操作系统对文件的访问、修改和保存为操作系统的文件管理系统
  • 文件属性：名称、标识符、类型、位置、大小、保护、时间、日期、用户标识
  • 文件操作：创建、写、读、重定位、删除、截断
  • 打开文件：文件指针、文件打开计数、磁盘位置、访问权限
  • 文件逻辑结构：
    • 无结构（流式文件：程序、目标代码）
    • 有结构
      • 顺序文件（记录连续存放，查找快，删除、修改复杂）
      • 索引文件（通过索引表查找逻辑文件）
  • 文件目录结构：
    • 概念：实现文件名到文件的映射，应该有可以重名、速度快的特点。
    • FCB（文件控制块）：实现“按名存取”文件的数据结构
      • 基本信息：文件名、文件物理位置、文件逻辑结构、文件物理结构
      • 存取控制信息：权限
      • 使用信息：建立时间、修改时间
    • 索引节点：一个FCB有64B，但在UNIX系统中一个目录项只占16B，其中14B文件名，2B存放索引节点指针，这样在检索文件时，其他文件描述信息不必加载到内存，加快了检索速度。
    • 目录结构：
      • 单级目录结构：整个文件系统中只有一张目录表
      • 两级目录结构：主文件目录、用户文件目录
      • 多级目录结构：如linux下以‘\’开头的文件目录
      • 无环图目录结构：实现文件共享
    • 文件共享：
      • 硬链接：文件的物理地址和其他信息不再存放在目录项中，而是索引节点中（linux下是inode节点），目录项只设置文件名和相应的指向inode的指针。inode有一个count记录指向它本身的指针数，如A和B指向了某一inode，A删除了该文件count-1，B删除后才真正删除
      • 软链接：快捷方式、符号链接、只存储路径名，不改变inode的count。
    • 文件访问：
      • 访问类型：读、写、执行、添加、删除、
      • 访问控制：拥有者、组、其他
    • 文件系统：
      • 类型：FAT32、NTFS、ext2
      • 包含：用户调用接口->文件目录系统->存取控制验证->逻辑文件系统与文件信息缓冲区->物理文件系统分配模块->设配管理程序模块
      • 目录实现：
        • 线性列表
        • 哈希表
      • 物理实现：
        • 连续分配
        • 链接分配
        • 索引分配：包含多层索引，如两层索引最大文件是4GB
      • 空闲链表管理：
        • 空闲表
        • 空闲链表法
        • 位示图发（0、1表示）
    • 磁盘管理
      • 地址结构：柱面号、盘面号、扇区号
      • 读写时间：寻道（磁头寻道）、延迟（磁头定位到扇区）、传输
      • 调度算法：先来先服务（根据进程请求顺序），最短寻道时间优先，扫描算法（电梯算法）、循环扫描

IO管理

  • IO分类：
    • 使用上分：
      • 人机交互：打印机、显示器、鼠标、键盘
      • 存储设备：光盘、磁盘、磁带
      • 网络：网络接口、调制解调器
    • 传输速度上分：
      • 低速：鼠标、键盘
      • 中速：打印机
      • 高速：磁盘、磁带、光盘
    • 信息交换单位分类：
      • 块设备：磁盘
      • 字符设备：打印机，特点是不可寻址
  • IO控制方式：
    • 程序控制方式：cpu利用率低
    • 中断驱动方式：io与内存数据交换要经过cpu，会消耗很多cpu
    • DMA（直接存储器存储）：io与内存直接交换数据，不经过cpu
    • 通道控制：对DMA的改进，把对一个数据块的cpu干预改为对一组数据块的cpu干预。
  • IO核心子系统：
    • 概念：包括IO调度、缓冲区和高速缓存、设备分配和回收、假脱机
    • IO调度：确定好的顺序来执行IO请求
    • 高速缓存：内存中存放磁盘数据
    • 缓冲区：硬件缓存或者存于内存
      • 单缓冲
      • 双缓冲
      • 循环缓冲
      • 缓冲池

测试HBase的调研

Veröffentlicht am 2018-01-18

使用HBase的背景：传统数据库的数据量越来越大

  • 单一数据库无法承受大量的并发访问，所以才有了数据库的读写分离，将数据库分为一个负责写的master库和多个负责读的slave库。因为分摊了 访问量，并且减少了锁的征用，可以减轻服务器压力。
  • 数据量增大的时候就要分库，将部相关的表分开部署，就需要分库，这样join就不能用了，需要借助中间层。
  • 数据量继续增大，分库也不能解决问题，就需要分表来解决，例如，将表的id%10存储在10个表中，这样源表有1000万的数据分表后每张表里只有100万的数据，减轻的数据库表的大小，提高TPS性能。

考虑到这些问题，HBase应运而生，对于大量的数据，只要加服务器的数量就行，HBase会自动水平切分扩展，跟hdfs的无缝集成保证了数据的可靠性和海量数据分析的高性能。应用场合就是可以实时读写超大规模的数据。

HBase架构：

  • 逻辑存储模型：

Hbase的表在逻辑上是一个稀疏表，该表也由行和列组成，但是每个列属于一个列族，行和列可以为唯一确定的存储单元叫元素。

  • 物理模型：

将逻辑模型中和行进行分割，按列族存储。每个表会分割成多个region，一个region由多个store组成，一个store存储所有列族的数据。一个store有一个memStore和多个StoreFile组成，Store是以HFile的方式存储在HDFS上，这也保证了数据的冗余存储，进而保证了数据的可靠性。

  • 高并发的读写原因：

使用日志文件和内存存储来将随机写改成顺序写（异步IO和缓存机制），保证稳定的数据插入速率，读写独立，没有冲突。

  • 物理组成：

HBase的物理组成：hdfs(NameNode+DataNode)+zookeeper(HQuorumPeer)+hbase(HMaster+HRegionServer)

现在HBase的应用场景：

  • 腾讯：HBase在几次营销活动（炫舞拉新活动、天天库跑新版预热活动、英雄联盟三周年活动等）中，从几十亿甚至几百亿条数据中实时拉取数据在毫秒级相应，成为营销活动的新渠道。 还有广告日志的处理，腾讯每天通过HBase处理百亿级的广告请求和曝光日志，访问HBase的延时80%在20ms内。此外，Hbase在其他点击交互日志或监控日志系统也有较多应用：如网络会话数据，秒级监控平台日志，微信支付日志等。
  • 阿里：2010年开始使用HBase，Hbase是 阿里的核心存储系统，它和计算引擎紧密结合，主要服务搜索和推荐的业务。
  • 其他应用：《HBase在智能电网异构数据同步中的应用》、《基于 HBase的工业日志系统设计与实现》、《基于 HBase 的旅馆业治安管理信息系统架构分析与设计》、《基于HBase的医疗卫生数据中心的构建与异构数据库同步的研究》

为什么要对HBase进行压力测试？

No SQL数据库在大数据处理领域得到越越多的应用，国内外学者也分别提出了对 No SQL 数据库进行性能测试的方法，然而针对 HBase 数据库进行性能测试的研究相对较少。

故障：2008 年，Amazon 的 S3服务出现长达近 8 个半小时的故障，导致全球部分地区用户无法访问其服务；2009 年，Google 公司的 GAE 出现严重故障，导致全球用户均无法正访问 GAE 网站。这些故障导致个巨大的损失。因此，在对于大数据库的压力测试在投入使用之前就显得特别重要。

ubuntu16.04通过odbc（用的C++）访问另一台centos上的达梦数据库

Veröffentlicht am 2017-11-14 | in 技术

背景

我的项目中需要C++访问达梦数据库，所以花了很大功夫来实现这个功能。
通过和达梦官方的邮件联系之后，得知现在达梦数据库（dm7）是不能在ubuntu版本的linux上安装server的，但是client还是可以安装在ubuntu上的。所以可以通过odbc来访问安装在centos上的达梦数据库。

服务端达梦的安装

在centos上安装的教程如下连接：http://blog.csdn.net/allens_zhou/article/details/8575400

ubuntu客户端配置：

首先要安装unixODBC，网上有教程；

然后安装soci，这是它的官网：soci.sourceforge.net/doc/3.2/，具体安装使用过程网上也有教程。

然后配置odbc manager和odbc driver

这个博客对我的帮助很大：http://www.itpub.net/thread-1824202-1-1.html，它指明了配置达梦odbc可以不需要libdmOdbcSet.so这个库，只要libdodbc.so就行了。具体操作见下：

  • 安装完odbc后，odbcinst –j 来查看安装位置等信息
  • 配置 odbcinst.ini，作用就是注册DM7的ODBC驱动,该驱动名称为DM7，内容如下：
    [DM7]
    Description = dm odbc
    Driver = /opt/dmdbms/bin/libdodbc.so
  • 配置 odbc.ini，作用就是创建系统DSN，名称为dm，调用名为DM7的ODBC驱动，内容如下：
    [dm]
    Description = dm7 dsn
    Driver = DM7
    SERVER =127.0.0.1
    UID = SYSDBA
    PWD = SYSDBA
    TCP_PORT = 5236
  • 配置完成之后，直接在终端运行：isql dm来测试，如果返回以下信息，测说明配置成功：
    [root@localhost opt]# isql dm
    +—————————————+
    | Connected! |
    | |
    | sql-statement |
    | help [tablename] |
    | quit |
    | |
    +—————————————+
  • 还可以建表，如create table student (id int, name char(128), score int)
  • 插入数据: insert into student (id, name, score) values (1,’myname’, 100)
